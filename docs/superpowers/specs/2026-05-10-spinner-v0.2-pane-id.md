# ccmux-spinner v0.2: explicit pane_id

- **Date**: 2026-05-10
- **Repo**: `ccmux-spinner`
- **Status**: design draft, awaiting user review
- **Targets**: ccmux-spinner v0.2.0 (breaking change)

## Revision history

- **2026-05-10 (initial)** — design driven by ccmux-core's need to
  bind a `SpinnerMonitor` to a specific pane it already has the id
  for (from claude-tap event envelopes), without the indirection of
  re-resolving from a tmux session name.

## Context

`ccmux-spinner` v0.1's `SpinnerMonitor(tmux_session)` internally
runs `tmux display-message -t <session>:0 -p '#{pane_id}'` to find
the pane it watches. This has two limitations:

1. **Hardcodes window 0.** The `:0` suffix means SpinnerMonitor
   always watches the pane that happens to be on window index 0 of
   the session. If the user does any tmux window manipulation
   (`new-window`, `select-window`, splitting active pane), the
   monitor can be watching the wrong pane.
2. **Couples pane resolution into SpinnerMonitor.** Callers that
   already know the pane id they want — and there are now real
   ones, starting with `ccmux-core`, which receives `pane_id`
   directly from `claude-tap` event envelopes — must still pass a
   tmux session name and let SpinnerMonitor re-resolve. Pointless
   indirection.

This spec inverts the contract: `SpinnerMonitor` takes a `pane_id`
directly; tmux-session-to-pane resolution moves into a separate
helper function that callers without a pane_id can use explicitly.

## Goals (v0.2)

1. `SpinnerMonitor` accepts `pane_id` instead of `tmux_session` in
   its constructor.
2. A new public helper `resolve_active_pane_id(tmux_session)`
   returns the **active** pane id for a tmux session (not "window
   0's pane"), with a fallback to window 0's first pane if no active
   pane is resolvable.
3. The existing `ccmux-spinner watch <tmux_session>` CLI is updated
   to call `resolve_active_pane_id` internally so its UX does not
   change.

## Non-goals (v0.2)

- **No new monitoring functionality.** Parser, overlay handling,
  change-coalescing, async-iterator mechanics: all unchanged.
- **No backward-compatibility shim.** v0.1 has no production
  consumers; a clean breaking change is cheaper than a
  permanent dual-arg signature.
- **No detection of mid-life pane moves.** SpinnerMonitor fixes the
  pane at construction. If the user moves Claude to a different
  pane mid-life, the monitor continues watching the original pane.

## Breaking changes

### `SpinnerMonitor.__init__`

```python
# v0.1 (removed):
SpinnerMonitor(tmux_session: str, poll_interval: float | None = None)

# v0.2 (new):
SpinnerMonitor(pane_id: str, poll_interval: float | None = None)
```

The internal call to `resolve_pane_id(tmux_session)` inside
`__aenter__` is removed; the constructor stores `pane_id` directly
and `capture_pane` calls use it as-is.

### `resolve_pane_id` is replaced by `resolve_active_pane_id`

```python
# v0.1 (removed):
def resolve_pane_id(tmux_session: str) -> str: ...
#     internally runs: tmux display-message -t <session>:0 -p '#{pane_id}'

# v0.2 (new):
def resolve_active_pane_id(tmux_session: str) -> str:
    """Return the active pane id of *tmux_session*.

    Resolution order:
      1. `tmux display-message -t <tmux_session> -p '#{pane_id}'`
         (no window suffix → tmux returns the active pane).
      2. Fallback: `tmux display-message -t <tmux_session>:0 -p '#{pane_id}'`
         (window 0's first pane), if step 1 produced no output or
         was rejected.

    Raises:
      TmuxResolutionError if tmux is missing, the session does not
      exist, or no pane could be resolved.
    """
```

The old `resolve_pane_id` is dropped (not aliased).

### Module-level changes in `pane.py`

- `capture_pane(pane_id: str)`: unchanged signature; still the
  pure subprocess wrapper.
- `resolve_pane_id`: removed.
- `resolve_active_pane_id`: new, replaces it.

### CLI

`ccmux-spinner watch <tmux_session>` is unchanged from the user's
perspective. Internally:

```python
# Was (v0.1):
async with SpinnerMonitor(tmux_session) as mon:
    ...

# Now (v0.2):
pane_id = resolve_active_pane_id(tmux_session)
async with SpinnerMonitor(pane_id) as mon:
    ...
```

## Migration

ccmux-spinner v0.1 has no production consumers. Internal call
sites are limited to the CLI and the test suite. No external
migration required.

After release, downstream packages (notably ccmux-core v0.1)
declare `ccmux-spinner >= 0.2.0` in their dependencies.

## Testing

### Unit tests (changes)

- **`resolve_active_pane_id`** — new test file
  `tests/test_resolve_active_pane.py` with mocked `subprocess.run`:
  - Active pane available → returns it.
  - Active pane lookup fails, `:0` fallback succeeds → returns
    `:0`'s pane.
  - Both lookups fail → `TmuxResolutionError`.
  - tmux binary missing → `TmuxResolutionError`.

- **`SpinnerMonitor`** — existing fixtures keyed by `tmux_session`
  refactored to pass `pane_id="%fake"` directly; the now-removed
  internal resolution step is no longer mocked.

- **CLI** — existing CLI test refactored to also mock
  `resolve_active_pane_id`.

### Drift / regression

The existing pane-text fixtures in `tests/` are unchanged; only the
construction wiring is updated. Parser, overlay handling, and
change-coalescing test coverage are preserved.

## Release plan

1. Branch `feat/v0.2.0-pane-id` off `main`.
2. Edit `pane.py`: replace `resolve_pane_id` with
   `resolve_active_pane_id` (try active, fall back to window-0).
3. Edit `monitor.py`: change `SpinnerMonitor.__init__` signature;
   remove the internal resolution call.
4. Edit `cli.py`: thread `resolve_active_pane_id` between the
   user-supplied tmux_session and the `SpinnerMonitor` construction.
5. Refactor existing tests; add new tests for
   `resolve_active_pane_id`.
6. Bump version to `0.2.0` in `pyproject.toml`, `_version.py`, any
   `tests/test_skeleton.py`. Update `CHANGELOG.md` with a clearly
   marked breaking-change entry.
7. Open PR; verify CI green.
8. Tag `v0.2.0` and push.

## Future work (deferred)

- **Following pane moves.** If consumers commonly hit "claude moved
  to a different pane" we can expose a `bind_pane_id(new_pane_id)`
  reset method, or have `SpinnerMonitor` accept a callable that
  returns the current pane id on each poll. Wait for real demand.
- **Pane-id rebinding helper.** A counterpart to
  `resolve_active_pane_id` that subscribes to tmux pane events
  (`-F` formatted stream of pane changes) instead of polling — only
  worth doing if polling cost becomes measurable.
